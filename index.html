1 多态

  • A:虚拟方法调用

    • 成员变量(属性)：
      • 编译看左边(父类)，运行看左边(父类)。
    • 成员方法：
      • 编译看左边(父类)，运行看右边(子类)。
      • 但是子类的特有方法无法调用（向下强转再调用），只能调用父类中定义过的方法，即只能调用父类中定义过的子类方法。
    • 静态方法：
      • 编译看左边(父类)，运行看左边(父类)。
      • (静态和类相关，算不上重写，所以，访问还是左边的)
    • 总结：
      • 只有非静态的成员方法,编译看左边,运行看右边

  • B: 实例：

    • Dog和Cat共同继承Animal类，Animal类中定义一个接收Animal对象的方法，
    • 当传入对象为Dog的实例时，animal.sleep()执行的是Dog实例的方法，而当传入对象为Cat的实例时，animal.sleep()执行的是Cat实例的方法
    • 对于子类特有对象则需要分别转换类型
  • C:
    多态是运行时行为，即编译期无法确定函数调用，只有在运行时才能确定

3 泛型

3.1 泛型方法&静态泛型方法

3.2 泛型通配符

  • 泛型通配符的含义是传递参数是选取符合条件类型中的一种
  • <?> 任意类型中的一种
    • 无限定通配符<?>很少使用，可以用替换，同时它是所有类型的超类
    • 无法使用add
  • <? extends E> 上界通配符
    • 为该实参E或其子类中的一种
    • 使用<? extend E>通配符作为方法参数，表示方法内部代码对于参数可以读，不能写。
    • 对于List，无法进行add操作:
      • 因为无法确定该泛型的实际参数，泛型下限也无法确定；假设B继承E，A继承B，若该泛型参数为子类类型A，根据规则可向其中存放不高于E类型的对象，往其中存放父类类型B，则明显不合适
  • <? super E> 下界通配符
    • 为该实参E或其父类中的一种
    • 使用<? super E>通配符作为方法参数，表示方法内部代码对于参数只能写，不能读。
    • 对于List，可以进行add操作:
      • 可以存放任何E(包含E)的子类对象，因为该泛型的下限确定为E，因而任何E的子类均可存放
  • 对比super和extend
    • <? extends T>允许调用读方法T get()获取T的引用，但不允许调用写方法set(T)传入T的引用（传入null除外）；
    • <? super T>允许调用写方法set(T)传入T的引用，但不允许调用读方法T get()获取T的引用（获取Object除外）。 - get()方法返回的是Object类型
  • 存取原则PECS
    • 如果你想从一个数据类型（Producer）里获取数据，使用 ? extends 通配符
    • 如果你想把对象写入一个数据结构里（Consumer），使用 ? super 通配符
    • 如果你既想存，又想取，那就别用通配符。

  • 和<?>

    • 是参数类型，常常用于泛型类或泛型方法的定义；
    • <?>是通配符，一般不能直接用来定义类或泛型方法，因为它不能直接参与操作，常常用于泛型方法的调用代码或泛型方法的型参。

git的配置

git仓库的初始化

git operation

  • git add readme.txt 添加
  • git commit -m [message] 提交
  • git status 查看状态
  • git diff 查看修改

已提交的版本修改

  • git log [—pretty=oneline] 查看提交历史，以便确定要回退到哪个版本
  • git log —graph —pretty=oneline —abbrev-commit 树状图展示
  • git reset —hard [HEAD^|commit id] 版本退回
  • git reflog 命令历史，以便确定要回到未来的哪个版本

PS: 在Git中，用HEAD表示当前版本,上一个版本就是HEAD^，上上一个版本就是HEAD^^，当然往上100个版本写100个^比较容易数不过来，所以写成HEAD~100

工作区的退回

将工作区的内容退回到最后一次git commit或git add后的版本

一种是文件自修改后还没有被放到暂存区，现在，撤销修改就回到和版本库一模一样的状态
一种是文件已经添加到暂存区后，又作了修改，现在，撤销修改就回到添加到暂存区后的状态
  • git restore
  • git checkout —

暂存区的退回

将git add的内容退回到工作区

  • git restore —staged
  • git reset HEAD

版本库的删除

  • git rm 将版本库中的文件删除
  • git commit 提交

工作区的a.txt删除了，可以使用git rm、git commit将文件从版本库删除；

若想恢复工作区的a.txt，可使用git restore将工作区的修改退回；

若已经使用git rm但没有提交，可使用git restore —staged 将暂存区的修改退回，再使用git restore将工作区的修改退回；

git推送

  • git remote add https://github.com/finyaci/learngit.git 加入远端仓库
  • git push -u master 第一次推送
  • git push master 之后推送
  • git pull master 从远程拉取仓库

git remote 操作

  • git remote -v 查看关联的远端仓库
  • git remote rm 删除远程库的名字

git分支操作

REF: https://www.liaoxuefeng.com/wiki/896043488029600/900003767775424

  • git branch 列出所有分支，当前分支前面会标一个*号。
  • git branch 创建一个分支
  • git [checkout|switch] 切换到指定分支
  • git switch -c 创建并切换到指定分支
  • git branch -d 删除指定分支
  • git merge 合并指定分支到当前分支

git stash

REF: https://www.liaoxuefeng.com/wiki/896043488029600/900388704535136

  • git stash 保存当前工作区内容
  • git stash list 查看保存列表
  • git stash pop 弹出工作区
  • git stash apply 恢复指定工作区不删除
  • git cherry-pick 将指定的提交复制到当前分支

修改bug过程

  • master上有bug，而此时你正在dev分支上工作，修改bug
  • 首先将工作区内容保存，其次从master创建分支issue-101修改bug，修改提交合并后，切换回dev分支，恢复工作区，并使用git cherry-pick命令将主分支上修改bug的内容也应用到dev分支上
